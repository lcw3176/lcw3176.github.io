Mapshot 서비스는 Netlify를 통해서, 이미지 API 서버는 Heroku를 통해 배포가 이루어지고 있습니다.
서비스 초창기에는 클라이언트 단에서 모두 처리가 가능하다고 판단, 정적 배포 사이트인 Netlify를 사용했으나
차후 기능이 확장되며 서버의 필요성이 생겼고, 배포와 관리가 간편하며 무료인 Heroku를 이용하게 되었습니다.
Google Analytics를 통해 사용자 데이터를 분석하고 있으며 Slack을 통해 에러 관련 메세지를 모니터링 중입니다.

사용자의 요청에 따라 각각 다른 방식으로 작동합니다.
네이버 지도를 이용한 배경 사진을 요청할 경우는 CORS 문제가 없기 때문에 클라이언트의 브라우저에서
지도 API에 알맞은 좌표의 사진을 요청, 이미지를 합친 후 파일로 제공합니다.
하지만 카카오 지도는 SOP 제한 때문에, 유저 브라우저에서 자체 가공이 불가능하고, Javascript를 통해 동적으로 지도가 생성됩니다.
그래서 Selenium을 통해 서버 내의 다른 컨트롤러로 이미지 생성 요청 후, 생성이 완료되면 화면을 캡쳐한 후에
이미지 파일을 바이트 코드로 Mapshot 서비스에 전송해 주는 방식으로 작동합니다.

사이트가 펑펑

서비스 초창기, 갑자기 유저에게 사이트가 작동을 안한다는 연락을 받았습니다. 이게 무슨 날벼락인가 싶어서 접속을 시도해 보았으나, 저 또한 접속이 되지 않았습니다. 급하게 이리저리 찾아본 결과, .tk, .ga등의 도메인은 한국에서 종종 접속 불량 이슈가 있었고, 해결 방법은 두 가지 정도가 있었습니다.

  1. 그냥 기다리기
  2. 클라이언트 DNS 서버 주소 변경하기
첫 번째 방법으로는 그냥 기다리는 것이었습니다. 그 동안의 사례들을 본 결과, 며칠 지나면 접속 이슈가 해결되는 경우가 일반적이었습니다. 두 번째 방법은 사용자 PC 환경의 DNS 서버 주소를 바꾸는 방법이었는데, 두 방법 모두 사실상 말이 안되는 해결책이었습니다. 결국 빠르게 도메인을 이전하는 방법을 택했습니다. 초창기 운영 방식은 netlify 서버를 tk 도메인에 연결해 놓았는데, 다행인 것은 netlify도 자체 도메인을 하나 제공해준다는 것이었고 서비스를 홍보했던 게시글의 주소를 netlify의 도메인으로 수정했습니다.

사용자 이전

이 시점에 남은 고민은, 여전히 tk로 접속하고 있던 유저들이었습니다. 사이트가 터진 당일날은 접속을 하지 않아서 netlify 주소로의 변경 사실을 모르거나, 아니면 다시 tk가 접속이 되니까 그대로 이용하는 유저들을 어떻게 netlify로 오게 할 것인지가 문제였습니다. 이때 몇 가지 방법을 생각했는데,

  1. tk 주소 폐쇄
  2. tk에서 netlify로 리다이렉트 걸어놓기
  3. tk 서비스 중지 후, 링크 남겨놓기
일단 1번은 너무 극단적이었습니다. 카페 홍보글을 통해 제 사이트를 인지한 사람이 아니라면 대체할 주소를 알 수단이 전무하다는 것이 문제였습니다. 2번 방법도 문제의 여지가 있던 게, tk 도메인이 작동하는 상황이면 아무 상관 없지만 기존에 tk 주소로 즐겨찾기를 해놨던 유저들은 tk 도메인이 또 터지면 리다이렉트도 무용지물이 되는건 마찬가지였습니다. 결국 제가 선택한 방법은 3번이었습니다. tk에 공지글 페이지 하나만을 남긴 채, netlify로의 유도하는 링크를 남겨 놓는 것으로 tk에서의 서비스는 마무리 되었습니다.

레이어는 말이없다

서비스가 한참 잘 운영되던중, 새로운 기능을 넣어보고 싶었습니다. 도시의 도로 윤곽만 있는 사진이라던가, 농업 지구, 계획 지구 등을 표시한 도시 계획 레이어 정보를 위성 사진에 포함하면 좋겠다는 생각이 들었고, 이 기능을 당장 넣어보기로 했습니다. 하지만 CORS 관련 에러가 떴고, 서버에서 조치를 취해주지 않으면 정적 웹페이지 서비스인 제 쪽에서는 사실상 할 수 있는게 없었습니다. 그런데 제가 이 기능 쓰자고 국가 관련 기관에 대뜸 설정을 바꿔달라고 할 수도 없고, 웹 서비스인데 브라우저를 안 거칠 방법도 없고, 머리가 아파서 저와 비슷한 상황의 사람들이 있었는지 해당 API 제공처의 문의 게시판을 둘러보았습니다. 역시나 비슷한 질문들이 있었고, 자체 프록시 서버를 구축하라는 대답을 확인했습니다.

개발만 한다고 될일이 아니구나

그래서 열심히 서버를 구축했습니다. 클라이언트가 원하는 정보의 옵션을 대신 전달해주고, 이미지 파일을 받아와 바이트로 리턴시키는 형식으로 만들었습니다. 레이어 서비스는 처음에는 Heroku라는 서버 호스팅 업체를 통해서 운영을 시도했습니다. 레이어 기능을 추가하고 잠이 든 날, 정말 이상하게도 문득 새벽 5시에 일어났고, 기분이 이상해서 레이어 기능을 테스트해봤습니다. 근데 이게 뭔일인지 동작을 안 했습니다. 곧 알게 된 사실인데, 해외 서버로의 한국 지리 관련 정보 반출은 전부 막혀있는 상태였습니다. 저는 급히 모든 기능을 롤백하고, 아침부터 다시 해결책을 찾기 시작했습니다.

분명 서울 리전인데?

이 때 발견한 것이 오라클의 무료 클라우드 서버였습니다. 무료로 인스턴스를 2개까지 발급이 가능했고, 서버 위치로 서울도 선택이 가능했습니다. 그래서 열심히 세팅 후 서버를 올려보았으나, 여전히 작동을 안했습니다. 서버 문제인가 싶어서 구글 블로그의 아무 이미지나 가져와 봤는데 매끄럽게 작동했고, 도대체 뭔 일인가 싶어서 아이피 위치추적 사이트에 들어가서 서버 아이피를 찍어보았습니다. 아이고, 국내에서 관리되는 아이피가 아니라고 나왔습니다. 그래서 다른 해외 서비스로 위치를 확인해보았는데, 위치는 분명 서울로 뜨고 있었습니다. 아마 국내 발급, 관리되는 아이피가 아니면 다 차단되는 듯 했고, 국내 업체인 네이버 클라우드로 다시 한번 서비스를 가동했습니다.

이런 로그는 필요없어요

네이버 서버 서비스 중에서도 Cloud Function 이라는 기능을 사용했는데, 간단하게 코드만 업로드 하면 굳이 리눅스 서버를 세팅할 일도 없거니와, 무료 제공량으로도 충분할 것이라는 판단에서였습니다. 그렇게 레이어 기능을 한 달 정도 운영했고, 저는 과금 폭탄과 함께 레이어 서비스를 종료했습니다. 분명 무료 사용량 만으로 충분했지만, 부가 서비스로 따라오는 Cloud Log Analytics라는 서비스가 있었는데, 해지도 안되고, Cloud Function 사용자에 한해서는 선택적으로 로그를 남길수도 없었습니다. 덕분에 저의 이미지 바이트 코드들이 왕창 서버에 로그로 쓸데없이 누적되었고, 여기서 돈폭탄을 맞게 되었습니다. 첫 기능 확장에 대한 꿈은 과금의 추억으로 끝났습니다.

눈 좀 뜨세요 헤로쿠씨

카카오 지도는 정적 지도 api를 제공하지 않아서, 자바스크립트로 렌더링 된 페이지를 셀레니움으로 캡쳐해서 mapshot 서비스에 제공해주는 식으로 작동하게 만들었습니다. 자잘한 문제도 많긴 했는데, 제일 굵직한 문제가 3개 정도 있었습니다.

  1. 30분 동안 요청이 없으면 서버 종료됨
  2. 30초 내에 요청을 처리해야함(헤로쿠 자체 정책)
  3. 위, 경도 값에 따라서 새 페이지 렌더링

첫번째 문제 해결!

먼저 첫 번째 문제부터 보자면,헤로쿠는 30분동안 요청이 없다면 서버를 종료시킵니다. 이 정책 때문에 카카오 지도를 서버가 잠든 이후 처음 호출하는 사람은 굉장히 긴 시간을 기다려야 했습니다. 이 문제는 비교적 쉽게 해결되었습니다. 먼저 처음 사용한 해결책은 외부 서비스 이용이었습니다. kaffeine이라는, 제 서버에 주기적으로 요청을 날려주는 서비스를 이용해봤는데, 서버가 종종 자는 모습이 보이길래 다른 방법을 찾아봤습니다. 두 번째 사용한 방법은 제가 제 서버를 10분마다 호출하는 것이었습니다. 그런데 저의 서비스 특성상 주말에는 접속이 거의 없는데, 쓸데없이 계속 켜놓는 기분이라 다른 방법을 다시 생각했습니다. 세 번째 방법으로는 Mapshot 서비스 방문 시 서버에 요청 보내기였습니다. 현재(22.02.03)까지 사용하고 있는 방법으로, Mapshot 사이트에 접속과 동시에 서버의 특정 컨트롤러에 요청을 날립니다. 서버를 깨우는게 목적이기 때문에 응답 값은 중요하지 않습니다. 만약 카카오 지도 기능을 이용한다면 빠른 응답이 가능하고, 이용하지 않아도 서버는 지도 관련 작업은 아무것도 하지 않기 때문에 큰 부하가 걸리지도 않습니다.

나머지 두개도 해결!

이제 남은 두 가지 문제가 있었습니다. 헤로쿠의 자체 정책상 30초 내에 요청을 처리하지 않으면 타임아웃으로 클라이언트와의 연결이 강제로 끊겼습니다. 게다가 헤로쿠 서버 스펙이 좋은 편이 아니라서, 요청이 들어오는 대로 이미지 캡쳐 관련 새 인스턴스를 만들면 메모리 부족, 타임아웃 등으로 인해 들어오는 요청 전부가 뻗어버렸습니다. 그래서 결국 사용된 방법이 일종의 대기 시스템이었습니다. 사이트에서는 지속적으로 내가 사용 가능한 상태인지 요청을 보내 확인하고, 사용 가능하다는 응답이 온다면 그 때 지도 이미지 생성 상태로 들어갔습니다. 하나씩 처리하는 속도는 비교적 빨랐기 때문에, 동시 접속자가 많지 않은 저의 서비스에는 부담이 없는 방식이었습니다. 4명까지 동시에 요청해도 1분 이내로 모두 처리가 완료되었고, 당분간은 크게 무리 없이 서비스가 가능할 것 같습니다.

소스 코드 관리

카테고리:
Development
문제:

서비스 장기화에 따른
코드 유지보수의 어려움

해결:

핵심 기능을 외부 라이브러리화,
UI 코드와 별개로 관리

스파게티 한 그릇 추가요

초창기 서비스의 코드는 아수라장 그 자체였습니다. 아무래도 자바스크립트로 뭔가 많이 짜본 적도 잘 없었고, 그에 따른 경험 부족으로 난장판의 코드가 펼쳐졌는데, 아마 UI쪽 제어 코드와 핵심 로직 파트의 코드가 모두 섞여 있는게 가장 큰 원인이었을 것입니다. 도저히 이런식으로는 장기적인 서비스 유지가 불가능 할 것이라고 판단했고, 계속해서 부분적으로 리팩토링을 해나갔습니다. 그러다 문득 든 생각이, 아예 핵심 기능들을 라이브러리로 만들어서 관리하면 내 사이트에는 UI관련 코드만 존재하게 될 것이고, 유지 보수에 훨씬 편하겠다는 판단이 들었습니다.

IE 멈춰

라이브러리를 만들다가 한 가지, 아주 큰 걸림돌이 있었는데 바로 Internet Explorer의 존재였습니다. 이놈은 클래스도 지원 안되고, 모듈화도 지원 안하고, 뭐 되는게 없었습니다. 그런데 사이트 접속자의 약 20%가 IE를 사용중이었기에, 이걸 다 포기하고 갈 수도 없는 노릇이었습니다. Babel 이라는 오픈소스가 저를 구원했고, 우여곡절 끝에 핵심 로직들을 분리했습니다. 이제 서비스의 유지, 보수가 어느정도 지속적으로 가능한 상태까지는 코드를 정리했습니다.

중요도와 흥미의 반비례관계

이렇게 정리된 핵심 기능들은 혹시 필요한 사람이 있을까봐, 중요 내용과 사용법들을 문서화 후 배포하고 있습니다. 물론 아무도 안쓰긴 합니다. 그래도 계속해서 업데이트도 하고 있고, 문서 시인성도 개선중인데, 한창 코드 짜다가 사용법 업데이트 할 시즌이 되면 그렇게 따분할 수가 없었습니다. 그래도, 언젠가 이 문서를 참고할 누군가 or 미래의 사용법을 까먹은 저를 위해 계속해서 업데이트 중입니다.

사이트 패치 타이밍

카테고리:
Infra
문제:

본 서비스 패치 진행 시간 결정

해결:

Google Analytics를 통해 축적된
유저 데이터 분석

근거가 필요하다

사이트 기능, 디자인을 패치할 일이 있을 때, 과연 이 패치를 언제 진행해야 좋을지 생각했습니다. 테스트서버에서 온갖 테스트를 겪고 올라온 코드들이지만, 본 서비스에 가면 가끔씩 에러가 나곤 했어서(주로 html 태그, id 등이 차이나서 생긴 문제) 되도록이면 서비스 이용도가 최대한 적은 시간에 패치를 진행하고 싶었습니다. 일단 든 생각은 업무용 사이트니까 그냥 단순히 주말에 패치하면 되지 않을까 였지만, 보다 확실한 근거를 마련하기 위해 구글 Analytics를 달았습니다. 이를 통해 일별 접속자, 시간때 별 접속자 통계를 확인할 수 있었습니다. 주로 평일 아침(9시 - 10시), 점심시간이 지난 오후 (1시 - 3시)가 제일 많았습니다. 예상대로 사용자가 가장 적은 시간은 주말이었고, 평일도 보통은 오후 11시 이후로는 접속자가 적어졌습니다. 그래서 보통 사소한 패치는 평일 새벽에 했고, 규모가 큰 패치는 주로 주말을 통해서 진행하게 되었습니다.

카테고리:

동기:

사이트에 대한 유저의 편한 피드백

해결:

간편한 문의 메일 기능 생성

서비스 개선점이 궁금하다

초창기 서비스에서는 개선점들이 많이 보였습니다. 그래서 하나, 둘 개선해 나갔는데, 문제는 이런 개선점들은 개발자인 저의 관점에서 본 불편함이었고, 실제 유저들이 어디서 불편함을 겪는지 궁금했습니다. 그래서 문의 메일을 텍스트로 따로 적어놨었는데, 생각보다 메일이 많이 오지는 않았습니다. 아무래도 업무용 사이트이다 보니, 다들 따로 메일을 보낼만큼 시간이 남지는 않을 듯 하여 사이트 내에서 메일을 보내는 기능을 넣었습니다.

피드백은 귀하다

문의를 받고, 서비스를 운영하며 느낀 것은, 실 사용자 중에서 기능 개선에 대한 건의나 궁금한 점을 묻는 분들은 생각보다 많지 않다는 것이었습니다. 도시공학 분야에 근무중인 친구를 통해 사석에서 업계인들에게 개선점을 들어본 적이 있었는데, 개선하면 좋을 점들을 아는 사람들은 많으나 실제 그 의견들이 저에게 전달되는 경우는 잘 없었습니다. 생각해보면 저도 어떤 서비스를 이용하면서 불편한 점이 있다면, 그냥 감수하고 이용하거나 그 서비스를 사용하지 않았던 적은 있어도, 구태여 개선점에 대해 뭔가 의견을 전달한 적은 없었던 것 같습니다. 서비스에 대한 피드백이 있다는 것은 저로서는 정말 감사한 일입니다.
